# JADX

After extracting an APK, one of the first questions is usually:

> _What does this application actually do?_

For most Android applications, the answer starts with **JADX**.

JADX is an open-source decompiler that converts Android DEX bytecode into readable Java source code. Instead of navigating low-level Smali instructions, JADX reconstructs a high-level view of the application's Java layer, making it much easier to understand its architecture and behaviour.

---

# DEX vs Java

An APK does **not** contain Java source code.

Instead, Android applications contain one or more **DEX** (Dalvik Executable) files.

```
Java / Kotlin

↓

Compiler

↓

DEX Bytecode

↓

APK
```

JADX works in the opposite direction.

```
APK

↓

classes.dex

↓

JADX

↓

Java (reconstructed)
```

The important word here is **reconstructed**.

The Java displayed by JADX was generated by the decompiler. It is not the original source code written by the developer.
For this reason, there is no point in modifying it!

---

# Before JADX

Before JADX became the standard Android decompiler, a common workflow consisted of converting DEX bytecode back into a Java archive.

```
APK

↓

classes.dex

↓

dex2jar

↓

classes.jar

↓

JD-GUI
```

**dex2jar** converts Dalvik bytecode into standard Java bytecode packaged as a JAR file.

That JAR can then be opened using traditional Java decompilers such as **JD-GUI**.

Although this workflow is still useful to know, JADX has largely replaced it by combining decompilation, resource browsing, Smali navigation and search capabilities into a single application.

---

# Why Use JADX?

JADX is often the fastest way to understand an Android application.

Within a few minutes you can usually identify:

- Activities
- Services
- Broadcast Receivers
- Fragments
- Network clients
- Database helpers
- Third-party SDKs
- Native library loading
- General application architecture

For many Android applications, this provides enough information to understand how the application works.

---

# Decompiled Does Not Mean Original

JADX does an excellent job reconstructing Java code, but it is still a **decompiler**.

Some information is permanently lost during compilation and cannot be recovered.

For example:

- Comments
- Formatting
- Original local variable names
- Some language constructs

Most of the time this makes little difference, but occasionally the generated Java may not exactly match the original source code.

When precision matters, Smali remains the authoritative representation of the DEX bytecode.

---

# Search Is Your Best Friend

One of JADX's most useful features is global search.

Instead of manually browsing packages, you can search directly for:

- URLs
- API endpoints
- Package names
- Method names
- UI strings
- Permission names
- Native library names

Searching is often much faster than navigating package trees manually.

---

# Typical Investigation

A common workflow using JADX might look like this:

1. Open the APK.
2. Inspect the package structure.
3. Locate the application's entry point.
4. Follow Activities and Fragments.
5. Search for interesting strings or URLs.
6. Identify native library loading.
7. Decide whether further investigation should continue in Java, Smali or native code.

As the investigation progresses, different tools naturally become more useful.

---

# Limitations

JADX focuses exclusively on the Java layer.

It cannot show:

- Native implementations
- Runtime memory
- Objects currently in memory
- Runtime method calls
- Dynamic behaviour

Some applications rely heavily on native code, making the Java layer relatively small.

In those situations, JADX remains useful for understanding how the application starts and how the Java layer interacts with native libraries, but additional tools become necessary.
